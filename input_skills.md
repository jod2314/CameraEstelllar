Lista de Habilidades Tecnológicas Priorizadas para el Proyecto en Android
🌟 Habilidades Críticas y Muy Importantes:
1.	Desarrollo Nativo en Kotlin/Java: Para la orquestación del ciclo de vida de la aplicación Android, diseño de UI y gestión de permisos del sistema.
Fase 1: Arquitectura Base y Control del Ciclo de Vida (MVVM y Servicios) Para orquestar una aplicación de captura continua sin agotar los recursos del dispositivo, el código Kotlin debe aislar estrictamente el hilo principal de la interfaz de usuario (Main Thread) de la canalización de la cámara. El hilo de la interfaz tiene un límite de tiempo de unos 16ms por cuadro para evitar tirones o bloqueos severos conocidos como errores ANR (Application Not Responding).
•	Implementación MVVM y Corrutinas: Debes estructurar la app utilizando el patrón Model-View-ViewModel (MVVM). Los objetos ViewModel persistirán los datos de la sesión de captura (como contadores de exposición o métricas de las estrellas) para que no se pierdan si el usuario rota el dispositivo o si ocurre un cambio en la configuración. Para las tareas asíncronas, el desarrollo nativo moderno en Kotlin debe usar Corrutinas (Coroutines) en lugar de callbacks anidados, ya que proporcionan una concurrencia estructurada, permitiendo un manejo de errores y cancelaciones más robusto.
•	Servicios en Primer Plano (Foreground Services): A partir de Android 9 (API 28), el sistema operativo bloquea el acceso a la cámara si la aplicación pasa a segundo plano. Dado que una ráfaga astrofotográfica puede durar minutos, la orquestación del ciclo de vida debe iniciar un servicio en primer plano para asegurar que la recolección de los cuadros RAW no sea "asesinada" por el sistema si la pantalla se apaga temporalmente.
Fase 2: Diseño de Interfaz de Usuario (UI) Nocturna El diseño de la UI debe ir mucho más allá de los botones convencionales, orientándose al uso en oscuridad total y a la asistencia predictiva.
•	Previsualización Nativa: La previsualización de la cámara debe inyectarse en componentes de bajo nivel del sistema Android, utilizando preferiblemente un SurfaceView o un TextureView enlazados al flujo del sensor.
•	Modo de Luz Roja (Red Light Mode): El diseño de la interfaz debe incluir obligatoriamente una máscara gráfica de luz roja. Esta característica filtra la luz azul y verde de la pantalla para evitar que el usuario pierda su visión escotópica (la adaptación natural del ojo a la oscuridad) durante el trabajo de campo.
•	Composición Predictiva mediante Sensores (AR): Utilizando el ciclo de vida de Kotlin, la UI debe suscribirse a los sensores de hardware del teléfono (giroscopio y magnetómetro) para construir una interfaz de Realidad Aumentada. Esto superpondrá gráficos vectoriales de constelaciones directamente sobre la previsualización de la cámara para guiar al usuario en la composición antes de disparar.
Fase 3: Gestión Rigurosa de Permisos y Almacenamiento Debido al volumen de datos (archivos RAW masivos) y la necesidad de registrar metadatos astronómicos exactos, el manifiesto y el control de permisos en tiempo real (Runtime Permissions) son críticos.
•	Permiso de Cámara e Integridad: Se debe requerir android.permission.CAMERA y, a nivel de manifiesto, configurar el requerimiento de la cámara como estricto si se desea bloquear la instalación en teléfonos no compatibles.
•	Permisos de Ubicación: Es indispensable solicitar permisos de ubicación fina (ACCESS_FINE_LOCATION) si el objetivo de la API es Android 5.0 o superior. Las coordenadas GPS son vitales en astrofotografía para incrustar el campo de metadatos estandarizados en las cabeceras de los archivos astronómicos (como AVM, EXIF o estándares para archivos FITS) y para poder calcular qué objetos celestes están visibles.
•	Almacenamiento (Scoped Storage): Guardar ráfagas de 30 fotos de 25MB requiere una lógica fuerte. Para Android 9 e inferiores, el permiso necesario es WRITE_EXTERNAL_STORAGE. Sin embargo, para Android 10 en adelante, Kotlin debe orquestar el guardado utilizando la API de MediaStore o el Scoped Storage de Android. Se debe apuntar al directorio público Environment.DIRECTORY_PICTURES con una subcarpeta personalizada, asegurando así que las valiosas imágenes astronómicas no se borren en caso de que el usuario desinstale la aplicación en el futuro.
Fase 4: Recuperación de Errores y Liberación del Hardware Un desarrollador nativo asume el control absoluto de las limpiezas de memoria. Si la app sufre un error, no puede dejar el hardware atascado.
•	Cierre Seguro del Hardware: La orquestación del ciclo de vida debe interceptar llamadas como onPause() y desvincular inmediatamente los recursos. Esto incluye llamar a CameraDevice.close() para liberar el módulo óptico de forma que el sistema operativo u otras apps puedan usar la cámara posteriormente. Si no se libera explícitamente, los siguientes intentos de apertura lanzarán bloqueos fatales.
•	Manejo Excepcional (Error Recovery): Todo el flujo de cámara en Kotlin/Java debe estar dentro de manejadores de excepciones tipo CameraAccessException y callbacks de estado como onError o onDisconnected. La lógica de UI debe presentar avisos claros e invocar la recuperación del estado si la cámara se satura o se desconecta por calor térmico durante tomas excesivamente prolongadas.

2.	Arquitectura Profunda de Camera2 API: Dominio en la creación asíncrona de CameraCaptureSession, configuración detallada de CaptureRequest por cada cuadro, manejo de CaptureResult y uso de DngCreator.
Módulo 2: Arquitectura Profunda de Camera2 API (Motor de Captura RAW)
Directriz General para la IA: El objetivo es construir un pipeline de captura asíncrono y de alto rendimiento utilizando Camera2 API. Este pipeline no debe usar algoritmos automáticos del ISP (Procesador de Señal de Imagen). Toda la captura debe priorizar el formato RAW_SENSOR para preservar la linealidad de los fotones. El código debe estar estrictamente desacoplado del Hilo Principal (Main Thread) utilizando corrutinas de Kotlin o un HandlerThread dedicado.
1. Creación Asíncrona de CameraCaptureSession e ImageReader
La sesión de captura debe configurarse para soportar un flujo masivo de datos sin agotar la memoria.
•	Configuración del Búfer (ImageReader): Se debe instanciar un ImageReader solicitando el formato ImageFormat.RAW_SENSOR. El tamaño del búfer debe coincidir con la resolución máxima del sensor (extraída de SENSOR_INFO_PIXEL_ARRAY_SIZE). El parámetro maxImages del ImageReader debe ser configurado con un valor óptimo (por ejemplo, 3 o 5) para mantener la ráfaga fluyendo sin lanzar una excepción de "búfer lleno".
•	Superficies de Salida (Targets): La sesión se debe inicializar utilizando CameraDevice.createCaptureSession(). Debe recibir al menos dos superficies: la superficie del visor (SurfaceView o TextureView para previsualización) y la superficie del ImageReader (para la captura RAW de alta resolución).
•	Gestión de Estado: Implementar un CameraCaptureSession.StateCallback asíncrono para manejar eventos onConfigured y onConfigureFailed de manera segura.
2. Configuración Detallada del CaptureRequest (El Bloqueo 3A)
Cada fotografía de la ráfaga debe ser matemáticamente idéntica a la anterior para que el apilamiento (stacking) funcione. La IA debe programar un CaptureRequest.Builder basado en el template CameraDevice.TEMPLATE_MANUAL.
La IA debe forzar la inyección de los siguientes parámetros en el CaptureRequest:
•	Apagado del Motor 3A: Establecer CaptureRequest.CONTROL_MODE en CameraMetadata.CONTROL_MODE_OFF. Esto desactiva Auto-Enfoque, Auto-Exposición y Auto-Balance de Blancos.
•	Exposición y Ganancia Constantes: Para evitar artefactos en la alineación, las ráfagas no deben usar exposición horquillada (bracketing). La IA debe inyectar directamente los valores calculados en CaptureRequest.SENSOR_EXPOSURE_TIME (en nanosegundos) y CaptureRequest.SENSOR_SENSITIVITY (valor ISO analógico).
•	Enfoque al Infinito: Configurar CaptureRequest.LENS_FOCUS_DISTANCE en 0.0f (que en la óptica de Android representa el infinito) o utilizar la calibración hiperfocal reportada por LENS_INFO_HYPERFOCAL_DISTANCE si está disponible.
•	Estabilización Desactivada: Forzar CaptureRequest.LENS_OPTICAL_STABILIZATION_MODE a OFF y CONTROL_VIDEO_STABILIZATION_MODE a OFF. En astrofotografía (con trípode), el OIS activo intenta compensar el movimiento de la Tierra de forma errática, arruinando las estrellas.
3. Orquestación del CaptureResult y Sincronización de Ráfagas
Enviar la ráfaga utilizando CameraCaptureSession.captureBurst() o setRepeatingRequest(). La IA debe implementar un mapeo estricto para conectar el resultado fotográfico con los píxeles reales.
•	Extracción de Metadatos (TotalCaptureResult): Al ejecutarse el callback onCaptureCompleted, la IA debe capturar el objeto TotalCaptureResult. Este objeto contiene los valores físicos reales que el hardware logró aplicar (que pueden variar ligeramente de lo solicitado en el CaptureRequest).
•	Sincronización Image - Result: Los píxeles llegan de forma asíncrona a ImageReader.OnImageAvailableListener, mientras que los metadatos llegan a CaptureCallback.onCaptureCompleted. La IA debe implementar una estructura de datos (como un TreeMap que use el timestamp o un Request ID como llave) para emparejar la imagen cruda exacta con su TotalCaptureResult correspondiente antes de procesarlos.
4. Uso de DngCreator para Empaquetado Científico
El fotograma no será útil para cálculos astrométricos si la IA no programa correctamente su empaquetado como un Archivo Negativo Digital (DNG).
•	Instanciación: Crear el DNG usando DngCreator(CameraCharacteristics, TotalCaptureResult). Es imperativo pasar el resultado específico de esa toma, y no características estáticas genéricas.
•	Metadatos de Calibración: Advertir a la IA que el uso de DngCreator asegura la incrustación automática de metadatos vitales para la astrofotografía en la cabecera del archivo, tales como:
o	SENSOR_CALIBRATION_TRANSFORM1 y SENSOR_COLOR_TRANSFORM1 (matrices para corrección de color estandarizada).
o	SENSOR_NOISE_PROFILE (coeficientes de ruido de lectura analógica S y O para perfiles de SNR).
o	SENSOR_DYNAMIC_BLACK_LEVEL y SENSOR_DYNAMIC_WHITE_LEVEL (niveles dinámicos para substracción de pedestal negro y rango dinámico real).
•	Escritura Fuera del Montículo (Off-Thread): La llamada al método dngCreator.writeImage(FileOutputStream, Image) es una operación pesada de Entrada/Salida (I/O). La IA debe orquestar esta escritura obligatoriamente en un hilo de fondo (ej. AsyncTask.THREAD_POOL_EXECUTOR o Dispatchers.IO en Corrutinas) y luego cerrar obligatoriamente el objeto Image con image.close() para liberar el búfer de vuelta al hardware. De omitirse este cierre, la API de la cámara se congelará al quedarse sin búferes disponibles.

3.	Desarrollo en C/C++ usando Android NDK: Indispensable para crear el motor de fotografía computacional que trabaje fuera de la recolección de basura del sistema Android y garantice velocidad en el manejo de archivos RAW masivos.
Directriz de Ingeniería para la IA: Módulo 3 - Motor de Fotografía Computacional en C++ (Android NDK)
Objetivo del Prompt: Construir el núcleo de procesamiento matemático de alto rendimiento en C++11/C++14. Este motor debe evadir por completo la recolección de basura (Garbage Collector) de la Máquina Virtual de Android (ART), procesando ráfagas masivas de archivos RAW en memoria nativa (Off-Heap). El código debe estar vectorizado y paralelizado para arquitecturas ARM.
-------------------------------------------------------------------------------- 
1. Puente JNI (Java Native Interface) y Transferencia "Zero-Copy"
La IA debe diseñar una interfaz JNI que no duplique la memoria. Copiar ráfagas de 30 fotos RAW (donde cada una puede pesar decenas de megabytes) desde Java hacia C++ agotará el montículo (Heap) y provocará un colapso.
•	Uso de DirectByteBuffer: La IA debe programar firmas JNI que reciban el búfer del ImageReader de Java como un ByteBuffer directo.
•	Punteros Crudos (Raw Pointers): En el lado de C++, la IA debe utilizar estrictamente la función env->GetDirectBufferAddress(buffer) para obtener un puntero a la memoria física unificada compartida entre la CPU y el hardware de la cámara, logrando lectura sin copias.
•	Pooling de Memoria Nativa: La IA debe implementar un administrador de memoria en C++ que preasigne grandes bloques de memoria al inicio de la sesión de captura (Bitmap Pooling). Esto evitará llamadas repetitivas a malloc() o new durante la ráfaga, previniendo la fragmentación.
2. Calibración Radiométrica y Preprocesamiento
Antes de alinear, los datos lineales crudos (10 a 14 bits empaquetados en arreglos de 16 bits) deben limpiarse de anomalías del hardware. La IA debe implementar kernels matemáticos para la calibración:
•	Sustracción de Ruido: El código debe aceptar Dark frames (cuadros oscuros) y Bias frames para restar el ruido térmico, ruido de lectura y "hot pixels" (píxeles calientes) del cuadro de luz principal (Light frame).
•	Corrección de Campo Plano: Implementar la división por Flat fields para corregir el viñeteado óptico y las sombras causadas por polvo en el sensor.
•	Fórmula Base: La IA debe codificar la corrección radiométrica iterando sobre los punteros de memoria bajo la fórmula general: Pixel_Final = (Pixel_Crudo - Dark) / Flat.
3. Motor de Alineación Estelar (Registration) mediante Asterismos
Dado que la Tierra rota, las estrellas se desplazarán entre cada captura de la ráfaga. La IA debe escribir un pipeline de visión computacional utilizando las librerías OpenCV y Eigen para C++.
•	Extracción de Centroides: La IA debe localizar las estrellas calculando su centro de gravedad (intensidad ponderada) con precisión sub-píxel.
•	Emparejamiento de Asterismos (Constelaciones): Los algoritmos genéricos de detección de esquinas fallan en astronomía. La IA debe construir triángulos de estrellas (asterismos) extrayendo los vecinos más cercanos de cada estrella.
•	Árboles kD (kD-Trees): La IA debe almacenar las longitudes de los lados de estos triángulos como puntos 3D en un árbol kD para buscar y emparejar rápidamente las constelaciones entre el cuadro de referencia y los cuadros en movimiento.
•	Cálculo de Matriz de Transformación: Una vez halladas las correspondencias de los asterismos, la IA debe usar RANSAC (Random Sample Consensus) para deducir la transformación afín u homografía y mapear los píxeles al cuadro de referencia.
4. Algoritmos de Apilamiento (Stacking) y Vectorización SIMD
El núcleo del valor científico de la app. La IA debe programar métodos estadísticos de superposición píxel por píxel para aumentar la relación señal-ruido (SNR), la cual mejora proporcionalmente a la raíz cuadrada del número de imágenes ( N  ).
•	Modos Requeridos: La IA debe exponer parámetros JNI para ejecutar los siguientes modos:
o	Average / Mean: Promedio matemático simple de la pila de imágenes para objetos de cielo profundo.
o	Median: Cálculo de la mediana para descartar anomalías temporales como aviones o satélites.
o	Kappa-Sigma Clipping: Algoritmo iterativo que calcula la desviación estándar (Sigma) y descarta estadísticamente los píxeles que se alejen demasiado de la media (rechazo de valores atípicos y rayos cósmicos).
o	Maximum Intensity: Conserva únicamente el valor del píxel más brillante para generar el efecto de trazos de estrellas (Star Trails).
•	Intrínsecas NEON (SIMD): La IA debe obligatoriamente vectorizar los bucles de suma de píxeles utilizando las instrucciones SIMD (Single Instruction Multiple Data) de la arquitectura ARM (NEON). Esto permitirá que un solo ciclo de CPU procese múltiples píxeles simultáneamente, reduciendo drásticamente el tiempo de compilación.
5. Módulo de Validación Metrológica (Calidad y Enfoque)
El código C++ debe ser capaz de evaluar la calidad de los datos en tiempo real y devolver métricas a la interfaz de usuario en Java/Kotlin.
•	Cálculo FWHM (Full Width at Half Maximum): La IA debe ajustar una curva gaussiana sobre el perfil de luz de las estrellas para determinar su nitidez, siendo un valor menor indicativo de un enfoque perfecto.
•	Relación Señal-Ruido (SNR): Implementar una rutina que compare la fuerza de la señal de los objetos contra la desviación estándar del fondo del cielo, permitiendo que la app decida algorítmicamente si la ráfaga ya tiene suficientes datos integrados.

4.	Programación en Vulkan (Compute Shaders): Capacidad para programar la GPU del dispositivo Android a bajo nivel para acelerar procesos como la corrección radiométrica, alineación de estrellas y superposición en paralelo.
Directriz de Ingeniería para la IA: Módulo 4 - Motor de Aceleración Gráfica con Vulkan (Compute Shaders)
Objetivo del Prompt: Construir el pipeline de renderizado y cómputo de bajo nivel utilizando la API Vulkan en Android. El objetivo principal es trasladar las operaciones matemáticas masivas (corrección radiométrica, alineación geométrica y apilamiento estadístico) desde la CPU hacia la GPU, reduciendo los tiempos de procesamiento de ráfagas RAW de decenas de segundos a milisegundos. El código debe evitar estrictamente los cuellos de botella de memoria mediante arquitecturas de "Zero-Copy" y aprovechar las micro-optimizaciones de las GPUs móviles (Adreno/Mali) como la precisión media (FP16) y la memoria compartida de los grupos de trabajo (Workgroups).
-------------------------------------------------------------------------------- 
1. Gestión de Memoria Unificada (Zero-Copy) y Evasión de Staging Buffers
El error más crítico al implementar Vulkan en móviles es utilizar el flujo de trabajo de PC tradicional, que implica copiar datos de la RAM a un Staging Buffer y luego a la VRAM, lo cual destruye el rendimiento y aumenta la latencia a niveles inaceptables (50-70ms por imagen). Dado que los SoCs móviles utilizan memoria unificada (la CPU y la GPU comparten la misma RAM física), la IA debe diseñar una arquitectura de importación directa:
•	Uso de AHardwareBuffer: La IA debe configurar el ImageReader de la Camera2 API para que asigne búferes respaldados por hardware (HardwareBuffer).
•	Extensión de Vulkan: El código en C++ debe utilizar la extensión VK_ANDROID_external_memory_android_hardware_buffer para mapear directamente estos búferes a objetos VkImage o VkBuffer de Vulkan sin realizar una sola copia en memoria.
•	Aliasing de Memoria: Para los búferes intermedios, la IA debe implementar la extensión VK_QCOM_tile_memory_heap (o equivalentes) para mantener los datos transitorios directamente en la memoria rápida de la GPU (GMEM) durante los pases de renderizado, minimizando el acceso a la memoria principal del sistema y ahorrando ancho de banda y batería.
2. Arquitectura del Compute Pipeline y Descriptor Sets
La IA no debe inicializar la canalización gráfica tradicional (Rasterización/3D), sino un Compute Pipeline puro y asíncrono.
•	Inicialización: El código debe instanciar un VkInstance, seleccionar el VkPhysicalDevice evaluando sus propiedades y crear un VkDevice solicitando explícitamente una cola de cómputo (VkQueue con capacidades de cómputo).
•	Layouts y Descriptores: La IA debe programar un VkDescriptorSetLayout que permita vincular múltiples búferes masivos simultáneamente: el búfer de la imagen RAW entrante, los búferes de calibración (Dark/Flat frames) y el búfer del acumulador (Stacking).
•	Push Constants: Para pasar parámetros ligeros que cambian por cada cuadro (como el número de imagen actual en la ráfaga, o las matrices de transformación geométrica 3x3), la IA debe utilizar Push Constants en lugar de actualizar Uniform Buffers, ya que son la vía de comunicación más rápida hacia el shader.
•	Sincronización: La IA debe separar estrictamente los envíos gráficos de los envíos de cómputo. El código debe orquestar dependencias de memoria usando vkCmdPipelineBarrier para asegurar que las escrituras del pase de alineación sean visibles antes de iniciar el pase de apilamiento.
3. Programación de los Compute Shaders (GLSL a SPIR-V)
La lógica astronómica se inyectará mediante pequeños programas que se ejecutan en los hilos de la GPU (Compute Kernels). La IA debe escribir estos shaders en GLSL (compilados offline a SPIR-V) implementando las siguientes micro-optimizaciones obligatorias para hardware móvil:
•	Precisión Media (FP16): La IA debe decorar las variables de píxel y las operaciones matemáticas utilizando mediump o explícitamente tipos float16_t a través de la extensión VK_KHR_shader_float16_int8. En GPUs móviles como Qualcomm Adreno, el uso de arquitectura escalar de 16 bits entrega el doble de rendimiento y consume la mitad de la energía comparado con los 32 bits (highp).
•	Alineación de Workgroups: El tamaño local del grupo de trabajo (local workgroup size) en el shader debe declararse explícitamente (ej. layout(local_size_x = 16, local_size_y = 16) in;) asegurándose de que sea múltiplo del tamaño del subgrupo (Subgroup Size, típicamente 32 o 64 en móviles) para maximizar la ocupación del hardware.
•	Memoria Compartida (Shared Memory): Para algoritmos que requieran examinar vecindarios de píxeles (como el filtro de la mediana o cálculos de varianza de ruido local), la IA debe cargar los bloques de la imagen global hacia la memoria compartida del hilo (shared float tile;), ejecutar barreras de sincronización (barrier();) y luego procesar.
4. Diseño de los Kernels Astronómicos Específicos
La IA debe proporcionar el código GLSL exacto para al menos tres pases fundamentales de la astrofotografía:
•	Kernel de Calibración Radiométrica: Un shader que reciba un Light frame RAW, le reste matemáticamente el Dark frame (o Bias) y lo divida por el Flat frame calibrado. La fórmula a implementar a nivel píxel es: Radiance = (Raw - Bias) / (Flat * Gain). Este proceso limpia el ruido del sensor antes de cualquier interpolación espacial.
•	Kernel de Reproyección (Alineación): Una vez que el código C++ en CPU calcula la matriz de transformación afín u homografía que empareja las estrellas, esta matriz se pasa al shader. El kernel debe reposicionar los píxeles del cuadro actual para que coincidan con el de referencia. La IA debe programar esto utilizando interpolación bilineal asistida por hardware (leyendo desde un VkImageView con un VkSampler configurado) para maximizar la precisión de las coordenadas fraccionarias de las estrellas.
•	Kernel de Apilamiento (Stacking & Clipping): El shader final debe fusionar todos los cuadros alineados. La IA debe programar un acumulador paralelo que lea el píxel actual y el histórico del búfer de acumulación. Debe incluir condicionales optimizadas (evitando divergencias de control de flujo graves) para implementar el Sigma-Clipping (rechazar píxeles que excedan una desviación estándar κ×σ, como satélites que cruzan la imagen) y luego aplicar un Average (promedio continuo) para multiplicar exponencialmente la Relación Señal-Ruido (SNR) del cielo nocturno.

5.	Gestión de Concurrencia en Android (HandlerThread / Corrutinas): Para administrar correctamente la captura a alta velocidad y aislar los procesos pesados de la interfaz del usuario.
Directriz de Ingeniería para la IA: Módulo de Gestión de Concurrencia (HandlerThread y Corrutinas)
Objetivo del Prompt: Construir una arquitectura de concurrencia robusta, libre de bloqueos y a prueba de fugas de memoria, diseñada específicamente para el procesamiento de ráfagas RAW masivas en la API Camera2. El código debe aislar estrictamente el Hilo Principal (Main/UI Thread) de los eventos de la cámara, y distribuir la carga matemática de la astrofotografía utilizando paralelismo nativo.
-------------------------------------------------------------------------------- 
1. Aislamiento Estricto del Hilo Principal (Anti-ANR)
La IA debe entender que el hilo principal (UI Thread) tiene la única responsabilidad de renderizar la pantalla cada 16 milisegundos (para lograr 60 fps). Si el procesamiento de un cuadro RAW bloquea este hilo por aproximadamente 5 segundos, el sistema lanzará un error fatal ANR (Application Not Responding).
•	Instrucción para la IA: Todo el ciclo de vida de la cámara debe delegarse a un HandlerThread dedicado (por ejemplo, nombrado "CameraBackground"). La IA debe instanciar este hilo, iniciar su Looper y generar un Handler en segundo plano.
•	Inyección del Handler: La IA debe inyectar este Handler de fondo de forma obligatoria en tres puntos críticos:
1.	En CameraManager.openCamera().
2.	En CameraCaptureSession.setRepeatingRequest() o captureBurst().
3.	En el listener ImageReader.setOnImageAvailableListener(). Esto garantizará que las matrices de píxeles masivas lleguen fuera del hilo de la interfaz de usuario.
•	Gestión de Prioridad: La IA debe configurar explícitamente la prioridad de este hilo usando Process.setThreadPriority(Process.THREAD_PRIORITY_BACKGROUND) para asegurar que el planificador de Linux no compita destructivamente con el renderizado de la UI.
2. Escalabilidad de Procesamiento Matemático (ThreadPoolExecutor y Corrutinas)
Mientras que un HandlerThread es perfecto para encolar secuencialmente los callbacks de la cámara, no es adecuado para tareas altamente paralelas, como aplicar reducciones de ruido por cada bloque de 8x8 píxeles de una imagen de varios megapíxeles.
•	Instrucción para la IA: Una vez que el HandlerThread recibe el objeto Image RAW, la IA debe derivar el procesamiento matemático pesado hacia un grupo de hilos. Para desarrollos modernos en Kotlin, se exige el uso de Corrutinas (Dispatchers.Default para matemáticas en CPU, y Dispatchers.IO para escritura de archivos DNG).
•	Alternativa Java (ThreadPoolExecutor): Si la IA programa en Java, debe instanciar un ThreadPoolExecutor que ajuste dinámicamente el número de hilos vivos según la carga de trabajo y el número de núcleos físicos del dispositivo, logrando ejecutar el apilamiento y la alineación estelar en paralelo. Se debe recordar a la IA que la CPU móvil solo puede manejar eficientemente un número limitado de hilos concurrentes.
3. Prevención de Fugas de Memoria (Memory Leaks) y Ciclo de Vida
Un problema crítico en las tareas asíncronas de larga duración (como ráfagas de astrofotografía) es la retención indebida de referencias de la interfaz de usuario. Si el usuario rota la pantalla, la Activity se destruye, pero un hilo de procesamiento podría seguir referenciando esa vista, provocando inestabilidad severa.
•	Bloqueo de Referencias UI: La IA tiene estrictamente prohibido pasar referencias directas o implícitas de objetos View o de la Activity hacia las corrutinas de procesamiento o al HandlerThread.
•	Patrón de Comunicación Seguro: La IA debe utilizar ViewModel combinado con LiveData (o StateFlow en Kotlin). El motor matemático solo debe actualizar estados en el ViewModel (por ejemplo, "compilando frame 15/30"). El hilo principal observará estos cambios y actualizará la UI en consecuencia, de forma reactiva y segura frente a los cambios del ciclo de vida (como rotaciones de pantalla).
4. Gestión Concurrente de Recursos Críticos (ImageReader y Punteros)
Al trabajar con un ImageReader que aloja archivos RAW masivos, existe un riesgo constante de colisiones: el sistema Android podría intentar cerrar el ImageReader cuando el usuario pausa la app, justo en el momento en que un hilo en segundo plano está extrayendo los píxeles de la última foto.
•	Instrucción para la IA (Conteo de Referencias): Para evitar excepciones al manipular la memoria de la cámara, la IA debe implementar un encapsulador de recuento de referencias (como una clase RefCountedAutoCloseable o similar mediante Mutex en corrutinas) para envolver el ImageReader de captura RAW.
•	Cierre Seguro: Esto garantiza que si el usuario interrumpe la toma, el ImageReader solo se destruya matemáticamente cuando todos los hilos de fondo activos hayan terminado de extraer y guardar las matrices de los archivos Image. Además, la IA debe asegurar que cada objeto Image crudo reciba siempre la orden image.close() al final de su procesamiento dentro del hilo correspondiente para devolver el búfer al hardware; de lo contrario, la cámara se congelará indefinidamente.

6.	Librerías Nativas de Visión Computacional (OpenCV para C++): Para la alineación robusta de estrellas y el cálculo de las matrices de transformación desde el código nativo.
Directriz de Ingeniería para la IA: Módulo 6 - Motor de Visión Computacional y Alineación Estelar (OpenCV en C++ NDK)
Objetivo del Prompt: Construir el módulo de visión computacional nativo utilizando OpenCV para C++ a través del Android NDK. El objetivo estricto de este módulo no es procesar la imagen final, sino calcular matemáticamente el desplazamiento de las estrellas entre el cuadro de referencia y los cuadros en movimiento. La IA debe programar un pipeline que extraiga características estelares con precisión sub-píxel, empareje asterismos (patrones de estrellas) y devuelva una Matriz de Homografía de 3x3 calculada mediante RANSAC. El código debe evitar la copia de memoria (Zero-Copy JNI) y estar optimizado para eludir las deficiencias de los detectores de esquinas genéricos en astrofotografía.
-------------------------------------------------------------------------------- 
1. Ingesta de Datos JNI y "Zero-Copy" con cv::Mat
La transferencia de un búfer RAW de 25MB desde Kotlin/Java hacia OpenCV en C++ es un punto crítico de fallo por memoria.
•	Instrucción para la IA: El método JNI debe recibir un DirectByteBuffer (el búfer RAW_SENSOR o un búfer de luminancia preprocesado) e instanciar un encabezado (header) de matriz de OpenCV sin copiar los datos subyacentes.
•	Código Requerido: La IA debe utilizar el constructor cv::Mat(int rows, int cols, int type, void* data) pasando el puntero obtenido con env->GetDirectBufferAddress(). Esto garantiza que OpenCV lea directamente de la memoria física de la cámara.
2. Preprocesamiento y Reducción (Bayer a Grises)
Alinear imágenes RAW completas de 12 o 48 megapíxeles en la CPU móvil es inviable.
•	Instrucción para la IA: Antes de buscar estrellas, el código en C++ debe reducir drásticamente la resolución de la imagen y convertir el patrón de mosaico Bayer a escala de grises.
•	Técnica de Submuestreo: La IA debe implementar un kernel optimizado (preferiblemente con intrínsecas SIMD/NEON) que promedie bloques de 2x2 píxeles Bayer (RGGB) para generar un único píxel en escala de grises. Esto no solo reduce la imagen a una cuarta parte de su tamaño (acelerando OpenCV), sino que actúa como un filtro paso bajo que mitiga el ruido electrónico y el aliasing inherente del sensor.
3. Detección de Estrellas y Precisión Sub-píxel (Centroiding)
Los algoritmos genéricos de OpenCV para detección de esquinas y características (como SIFT, SURF o FAST) suelen fracasar en imágenes astronómicas porque las estrellas carecen de estructura interna distintiva (son solo "manchas" blancas sobre fondo negro).
•	Aislamiento de la Señal: La IA debe aplicar un umbralizado (Thresholding) u operaciones morfológicas (como Difference of Gaussians o cv::threshold basado en la desviación estándar del fondo) para aislar las fuentes puntuales de luz.
•	Centro de Gravedad Ponderado (IWC): Una vez aisladas las "manchas" (blobs), la IA no debe conformarse con las coordenadas enteras del píxel más brillante. Debe programar una función que calcule el Centro Ponderado por Intensidad (Intensity Weighted Center) de cada estrella. La fórmula a implementar es xcog=∑Ii∑(xi×Ii), donde Ii es la intensidad del píxel. Esto otorga coordenadas en punto flotante (sub-píxel), vitales para la alineación perfecta del apilamiento.
4. Emparejamiento por Asterismos (Constelaciones)
Alinear mediante descriptores visuales (como ORB) es difícil en astronomía profunda. El método estándar de oro ("Astroalign") es el emparejamiento geométrico de asterismos.
•	Construcción de Triángulos: La IA debe tomar las coordenadas sub-píxel de las ~50 estrellas más brillantes del cuadro y generar triángulos (asterismos) combinando cada estrella con sus vecinas más cercanas.
•	Invariantes Geométricos y KD-Trees: Para cada triángulo, la IA debe calcular las longitudes de sus tres lados y ordenarlas. Estos tres valores formarán un punto en un espacio 3D.
•	Búsqueda Rápida (Flann / KD-Tree): La IA debe utilizar el módulo cv::flann (Fast Library for Approximate Nearest Neighbors) o un KD-Tree de OpenCV para buscar eficientemente qué triángulo del Cuadro B tiene las dimensiones más idénticas al triángulo del Cuadro de Referencia A. Los vértices de los triángulos emparejados exitosamente se convertirán en los puntos de control (std::vector<cv::Point2f>).
5. Cálculo de la Matriz de Transformación (Homografía + RANSAC)
Con los puntos de control (estrellas) emparejados entre ambos cuadros, el último paso de OpenCV es deducir el movimiento exacto de la Tierra o el temblor de las manos.
•	Homografía (8 Grados de Libertad): Aunque las estrellas están en el infinito, un simple desplazamiento (transformación rígida) suele ser insuficiente si hubo ligera rotación de campo o distorsión de perspectiva en lentes gran angulares. La IA debe instruir la creación de una matriz de proyección de perspectiva 3x3 (H).
•	Rechazo de Valores Atípicos (RANSAC): Es crucial que la IA utilice la función cv::findHomography(points_A, points_B, cv::RANSAC). Al usar RANSAC (Random Sample Consensus), OpenCV descartará matemáticamente cualquier emparejamiento erróneo (por ejemplo, confundir un hot-pixel o un satélite con una estrella) y devolverá la matriz de transformación más óptima.
•	Salida del Módulo: El código C++ no debe aplicar la deformación (cv::warpPerspective) sobre la imagen RAW si el dispositivo tiene soporte para Compute Shaders en Vulkan, ya que la CPU será un cuello de botella. El output de este módulo de OpenCV a través de JNI será única y exclusivamente el array de 9 floats (Matriz 3x3) de la Homografía calculada. Este array será el que posteriormente inyectarás a la GPU (Vulkan) mediante Push Constants para que renderice la imagen a velocidad de hardware. (Nota: La IA puede programar un bloque cv::warpPerspective solo como plan de contingencia para procesadores sin soporte Vulkan).
